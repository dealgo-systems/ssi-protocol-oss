## 1. Executive Overview

Phase X extends the substrate from internal governance correctness into **cross-platform operational trust**: the property that a governance decision, its operator approvals, its external execution, and its outcome remain a single verifiable lineage — even when execution crosses into third-party infrastructure (irreversible payments, deployment platforms, autonomous agent runtimes).

### What Phase X Delivers

| Capability | Before Phase X | After Phase X |
|---|---|---|
| Governance scope | Internal decisions + receipts | Decisions whose execution lands in external infrastructure |
| Continuity boundary | Single process / single chain | Cross-platform lifecycle (request → approval → external exec → outcome → receipt) |
| Evidence portability | Internal receipts, offline verifiable | Externally attestable evidence packets, regulator-grade lineage |
| Integration discipline | Ad-hoc | Target qualification + boundary classification + tier model |
| Failure semantics | Per-component | Standardized continuity failure taxonomy |

This document is the **sequencing authority** for X-A (irreversible financial), X-B (deployment), and X-C (autonomous tool) sub-phases. No sub-phase code begins until the full set of §7 production readiness gates is met.

## 2. Mission & Interoperability Thesis

**Mission.** Prove that verifiable operational continuity survives platform boundaries.

**Thesis.** AI systems and human operators increasingly drive irreversible action across third-party infrastructure. Today, each platform produces its own evidence in its own schema; the *connective lineage* — who decided, on what policy, with whose approval, with what outcome — is reconstructed manually, if at all. The substrate already produces that lineage internally. Phase X extends it across the boundary so that an irreversible payment, a production deployment, and an autonomous tool call all share **one verifiable governance narrative** with portable, offline-verifiable receipts.

The product is **cross-platform governance continuity**. External integrations are proving grounds, not the product.

## 3. Trust Invariants

These MUST NOT break under any sub-phase:

1. **Substrate authority preservation.** The substrate's v2.0 invariant — *"v2 can influence. v1 decides."* — is preserved across every Phase X surface. No Phase X mechanism grants v2 standalone decision authority. v2 emits bias and observation; v1 (or the equivalent decision authority in any platform extension) decides.
2. **Byte-identical compatibility.** Callers that do not opt into Phase X behavior MUST receive byte-identical decisions to the v2.0-frozen substrate. Phase X is purely additive.
3. **Receipt continuity.** Every externally-executed action attaches to a receipt whose hash is committed into the governance chain before or at execution.
4. **Hash-chain monotonicity.** Chain heads only advance; rewriting history is detectable by any holder of a prior receipt.
5. **Operator identity binding.** Every approval carries an Ed25519 signature over canonical decision bytes; replay-bound by nonce + timestamp.
6. **Frozen policy snapshot.** The policy version that gated a decision is captured by content hash at decision time and embedded in the receipt.
7. **Replayability.** Given a receipt + frozen policy snapshot + input, the decision is deterministically reproducible.
8. **Offline verifiability.** Receipts verify without contacting any first-party service.
9. **No silent external execution.** Any external action without a corresponding receipt is a Tier-violation incident (see §12).

## 5. Governance Flows

### 5.1 Canonical Lifecycle

```
agent or human request
  → policy evaluation (frozen snapshot captured)
  → escalation decision
  → operator approval (Ed25519-signed)
  → pre-execution receipt committed to chain
  → external execution
  → external outcome ingested (webhook or polled)
  → outcome attached to receipt; chain head advances
  → offline verifier confirms full lineage
```

### 5.2 X-A — Irreversible Financial Governance

```
agent: "refund $X for charge <id>"
  → policy: amount tier, refund-policy snapshot v_hash
  → escalation: irreversible-action class → operator approval required
  → operator approves (signed)
  → pre-exec receipt R₁ committed
  → external API call (idempotency_key = R₁.id)
  → provider webhook → outcome bound to R₁
  → R₁ finalized; chain head advances
  → evidence packet exportable
```

### 5.3 X-B — Deployment Governance

```
agent or human: "promote change <ref> to production"
  → policy: deploy-risk snapshot, change-class evaluation
  → escalation by risk tier
  → operator approval(s) collected
  → pre-exec receipt
  → external merge/deploy (idempotency via commit/deploy ID)
  → deployment outcome (success/failure/rollback) attached
  → chain head advances
```

### 5.4 X-C — Autonomous Tool Governance

```
agent invokes tool with side-effects
  → substrate evaluates tool call against policy snapshot
  → escalation if tool is in dangerous-class registry
  → operator approval (or auto-approve under tier)
  → pre-exec receipt
  → tool executes; structured outcome captured
  → outcome attached; chain advances
```

## 7. Production Readiness Gates

Before sub-phase code merges:

1. **Producer key rotation registry** — versioned signing keys, rotation events themselves chained. (The v2.0-frozen substrate spec explicitly defers production-grade rotation to v2.1; this gate fulfills that.)
2. **Frozen policy snapshots** — content-hashed, retrievable by hash, embedded in receipts. Per-call policy override (where supported) is reflected in the recorded snapshot hash.
3. **Receipt schema stabilization** — schema versioning + canonical-byte rules locked (see §14). Reference RPX schema in this repo evolves under additive-only discipline.
4. **Cross-language verifier** — at least one verifier outside the primary runtime (e.g., a Python sibling alongside the existing TypeScript `ssi-verify`) reproducing chain validation against the same golden vectors.
5. **Evidence packet export** — single command produces auditor-consumable bundle.
6. **Webhook surface hardening** — every inbound external webhook is HMAC-verified, replay-protected, and idempotency-keyed against the receipt id.

## 8. Threat Model

| Threat | Mitigation |
|---|---|
| Operator approval forgery | Ed25519 signature over canonical bytes |
| Webhook replay | HMAC + nonce/timestamp + idempotency key bound to receipt |
| Webhook spoofing | Provider-specific signature verification + allowed source pinning |
| External execution without receipt | Idempotency keys = receipt ids; reconciliation job flags orphaned external state |
| Policy drift between decision and execution | Frozen policy snapshot embedded in receipt |
| Cross-platform desync (chain says X, external says Y) | Outcome reconciliation, continuity-failure incident class |
| Tampering with stored receipts | Hash-chain monotonicity + offline verifier |
| Operator abuse (legitimate approvals used outside policy intent) | Approval scoped to receipt id; analytics on per-operator approval patterns |
| Producer key compromise | Key rotation registry; receipts older than rotation remain verifiable under prior key |
| Substrate authority drift | Invariant 1 + invariant 2 enforced in every Phase X test suite |

## 12. Operational Continuity Failure Modes

Standardized incident classes. Every Phase X incident MUST map to one of these:

1. **Receipt mismatch** — chain receipt content disagrees with stored canonical bytes.
2. **Webhook replay** — same external event ingested twice, idempotency check fired.
3. **Operator-action gap** — approval present but no execution attached within SLA.
4. **External execution without receipt** — external state shows action; no corresponding receipt or pre-exec receipt.
5. **Policy drift** — frozen snapshot hash does not resolve, or resolves to different bytes.
6. **Frozen snapshot mismatch** — receipt embeds snapshot hash that does not match retrieved policy bytes.
7. **Cross-platform desync** — internal chain head and external system of record disagree on terminal state.
8. **Orphaned outcome** — outcome event with no parent receipt.
9. **Unverifiable remediation** — remediation action not itself receipted.
10. **Substrate authority violation** — any code path through which v2 (or its Phase X extension) takes a final decision authority that belonged to v1. Severity floor: critical.

Each class has a fixed severity floor and remediation runbook (defined per sub-phase).

## 14. Receipt Compatibility & Evolution

Receipt schema is a public protocol surface. Rules:

1. **Additive-only fields.** New fields are optional; verifiers ignore unknown fields. No field is ever removed or repurposed.
2. **Schema freeze rules.** Major version increment requires new verifier release that accepts both versions.
3. **Canonical-byte preservation.** Canonicalization rules (field ordering, normalization, encoding — RFC 8785 in this repo's reference implementation) are locked at v1; any change is a major-version event.
4. **Deprecation.** Fields are deprecated, never deleted; deprecated fields remain canonical for hashing.
5. **Verifier backward compatibility.** Latest verifier MUST verify all prior receipt versions still in chain.
6. **Producer key rotation** is a chained event itself — receipts pre-rotation remain verifiable under prior public key advertised in the registry.

## 15. Target Qualification Rules

Before any new platform integration enters Phase X, it MUST pass this checklist:

- [ ] Does the platform expose at least one **irreversible** or **slow-reversible** action class worth governing?
- [ ] Does it provide **webhooks** (or pollable outcome events)?
- [ ] Is **external evidence** (transaction id, SHA, deployment id, tool call id) retrievable post-hoc?
- [ ] Are external IDs **deterministic / idempotency-key-able** so receipt id can bind execution?
- [ ] Is the action **replayable** in a test/sandbox mode?
- [ ] Is there a **policy insertion point** (pre-execution hook, approval API, request signing)?
- [ ] Is **operator escalation** feasible inside the substrate before execution?
- [ ] Is **offline verification** of the ingested outcome feasible (signed webhook or signed receipt from provider)?

A target failing two or more boxes is rejected. A target failing one is documented as a known limitation in its sub-phase spec.

## 17. Verification

Doctrine adoption is verified per sub-phase by producing:

1. **Live demo** — end-to-end canonical flow against real (test-mode) external infrastructure.
2. **Receipt** — signed, chain-committed, with frozen policy snapshot.
3. **Offline verifier run** — independent process re-validates the chain segment, using at minimum the reference verifier in this repo (`tools/ssi-verify/`).
4. **Lineage screenshot** — operator → approval → execution → outcome lifecycle graph.
5. **Evidence packet** — exportable bundle suitable for auditor handoff.
6. **Failure-mode drill** — at least one §12 failure class deliberately triggered and remediated, with the remediation itself receipted.
7. **Substrate-invariant regression** — invariants 1 and 2 (substrate authority + byte-identical compatibility) MUST be re-verified against the substrate's own acceptance test suite after any Phase X change.

## 18. Framing Discipline

Throughout all Phase X work, the product is described as **cross-platform governance continuity** — never as a list of named platform integrations. External infrastructure targets are *proving grounds*. The doctrine is the product surface; the receipts are the evidence; the chain is the substrate; the verifier is the trust-transfer mechanism.

If an internal artifact, slide, or PR description starts framing Phase X as integrations, that is a doctrine drift incident and should be corrected at review.

## 19. Substrate Lineage

Existing substrate primitives this doctrine inherits:

| Primitive | Substrate origin |
|---|---|
| Ed25519 producer-key registry (env-loaded) | substrate v2.0 — explicitly named for v2.1 hardening |
| Signed observations layer (signature + chain + writer + reader) | substrate v2.0 |
| Hash-chain continuity for observations | substrate v2.0 |
| Byte-identity verification scripts | substrate v2.0 |
| Signed outcome ingestion channel | substrate v2.0 |
| RPX record schema + canonical-byte rules + reference verifier | this repo |
| Receipt bundle assembly + offline browser/Node verifier | the portal repo |

Phase X composes these into cross-platform continuity. It does not replace them, fork them, or re-implement them.
